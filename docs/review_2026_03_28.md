# Branch Review: `feature/fix-rounding-vat-rates` (2026-03-28)

## Scope

This review validates the current branch against the findings/plan described in `docs/chnagelog_2026_03_28.md`.

Reviewed files (branch diff vs `origin/main`):

- `src/wexbo2pohoda/wexbo2pohoda.py`
- `tests/test_unit.py`
- `tests/test_integration.py`

## Summary (what changed)

- Added `round_price()` to round to **1 decimal place** using `Decimal(..., ROUND_HALF_UP)`.
- Added `get_vat_items()` to extract **multiple VAT buckets** from one WEXBO invoice item:
  - `high` (21%) via `price_high` / `price_vat_high`
  - `low` (12%) via `price_low` / `price_vat_low`
  - `none` (0%) via `price_none` / `price_vat_none`
- Updated `create_invoice_items()` to create **one `invoiceItem` per VAT bucket** and set:
  - `rateVAT` according to the bucket
  - `text` according to the bucket
  - `unitPrice` to the rounded base price
- Added extensive unit + integration tests for rounding and multi-VAT behavior.

## Comparison vs plan in `docs/chnagelog_2026_03_28.md`

### 1) Missing multiple VAT rates support

- **Plan**: export multiple items when invoice contains different VAT rates (e.g. invoice `32600049` has both `high` and `none`).
- **Implemented**: `get_vat_items()` returns multiple bucket entries and `create_invoice_items()` creates multiple `invoiceItem`s.
- **Status**: matches plan.

### 2) Rounding issues

- **Plan**: implement rounding to 1 decimal with HALF_UP semantics.
- **Implemented**: `round_price()` uses `Decimal(str(value)).quantize(Decimal('0.1'), ROUND_HALF_UP)`.
- **Status**: matches plan.

### 3) Books / low VAT

- **Plan**: support reduced VAT stored in `price_low` (12%).
- **Implemented**: `get_vat_items()` includes `low` and uses a 1.12 fallback multiplier if `price_vat_low` is missing.
- **Status**: matches plan.

### 4) “Use `price_vat_*` values” to avoid Pohoda recalculation

- **Plan intent**: mitigate Pohoda recalculation differences by using the precomputed `price_vat_*`.
- **Implemented**:
  - `get_vat_items()` *does* compute and return `price_with_vat` (preferring `price_vat_*` when present).
  - However, the generated Pohoda XML currently only outputs `unitPrice` (base price) + `rateVAT`.
- **Status**: partial.

In other words: **the code calculates `price_with_vat` but does not write it into the output XML**, so Pohoda will still derive totals from base+VAT rate. The current approach relies on rounding base prices to reduce discrepancies.

## Review findings / remaining problems

- **[Gap vs changelog verification section]**
  - The changelog claims invoice `32600049` “After (correct)” includes total-with-VAT `10,286`.
  - The current integration test `test_invoice_32600049_total` verifies **base price sum** (`7990.1 + 618 = 8608.1`), not the final “with VAT” total.
  - If the acceptance criterion is “Pohoda total equals WEXBO total with VAT”, we still lack an automated test that validates the *with-VAT* total based on how Pohoda interprets the XML.

- **[Computed `price_with_vat` is unused]**
  - `get_vat_items()` returns `price_with_vat`, but `create_invoice_items()` does not include it in the output XML.
  - If the goal is to force exact final totals from the source XML, consider exporting fields that represent total-with-VAT / VAT amount (depending on the Pohoda schema).

- **[Formatting consistency]**
  - Output uses `str(float)` which yields values like `1000.0`.
  - If Pohoda import or downstream systems require a specific string format (e.g. fixed `1` decimal or `2` decimals), it’s safer to format explicitly (e.g. `f"{value:.1f}"`).

## Recommendations / next steps

- Confirm the intended contract for Pohoda import:
  - **Option A**: base price + VAT rate is sufficient, and rounding to 1 decimal is the intended fix.
  - **Option B**: the export must carry exact with-VAT amounts from WEXBO (`price_vat_*`) to fully prevent Pohoda recomputation differences.

- If Option B is required:
  - Identify the correct Pohoda XML elements for “price with VAT” and/or “VAT amount” for `invoiceItem`, and write both base and with-VAT values (or disable recalculation if supported).
  - Add a targeted integration test that parses the produced XML and verifies those exported amounts for invoice `32600049`.
